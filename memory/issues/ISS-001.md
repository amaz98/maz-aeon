# ISS-001 — Recurring skipped heartbeat dispatch slots

## Observation

`heartbeat` is the only enabled skill. It is scheduled 3× daily at 08:00 / 14:00 / 20:00 UTC via `aeon.yml`. Over the last 3 days (2026-06-29 → 2026-07-01), 5 of 9 scheduled slots have failed to dispatch:

| Slot | Dispatched? |
|------|-------------|
| 2026-06-29 08:00 | ❌ skipped |
| 2026-06-29 14:00 | ❌ skipped |
| 2026-06-29 20:00 | ✅ ran at 20:25 |
| 2026-06-30 08:00 | ✅ ran at 08:19 |
| 2026-06-30 14:00 | ❌ skipped |
| 2026-06-30 20:00 | ✅ ran at 20:20 |
| 2026-07-01 08:00 | ❌ skipped |
| 2026-07-01 14:00 | ❌ skipped |
| 2026-07-01 20:00 | ✅ ran at 20:15 (this run) |

Success rate over window: **4/9 = 44%**.

The heartbeat workflow itself always completes successfully once dispatched (`total_runs=92`, `total_successes=92`, `consecutive_failures=0` in `cron-state.json`) — the problem is that GitHub Actions is failing to fire the cron on schedule.

## Escalation trigger

Yesterday's heartbeat log (2026-06-30 20:20 UTC) explicitly committed to filing an issue if the 2026-07-01 08:00 slot also skipped. It did — plus the 14:00 slot after it. Escalation condition met.

## Suspected root cause

Public GitHub Actions scheduled workflows are known to be dispatched on a best-effort basis and can be delayed or skipped during high-load periods, especially on free tier. Reference: https://docs.github.com/en/actions/using-workflows/events-that-trigger-workflows#schedule.

This is likely not a bug in the repository — but the recurrence rate (5/9) suggests either:
1. A capacity issue on GitHub's side for this repo/account
2. A workflow-file misconfiguration that causes the scheduler to intermittently skip
3. Repo inactivity throttling (public repos with no recent commits may deprioritize scheduled runs; note the last non-heartbeat commit was `2b51b8d chore(cron): heartbeat success`)

## Next steps (proposed)

- Add a lightweight redundant trigger — e.g. a second cron entry at 08:05 / 14:05 / 20:05 UTC, or a `workflow_dispatch` fallback the operator can fire manually
- Consider running the scheduler on a self-hosted runner or via an external ping (e.g. GitHub-external cron service that hits `workflow_dispatch` via the API)
- Verify the workflow file has no syntax issues that could cause silent skips
